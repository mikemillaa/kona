# P2P Scoring Configuration Example

This comprehensive example demonstrates how to configure peer and topic scoring for kona-p2p networks. It provides practical patterns for implementing scoring in different network scenarios, from testing environments to production deployments.

<Callout type="info">
This example showcases the `example-p2p-scoring` project located in the [examples directory](https://github.com/op-rs/kona/tree/main/examples/p2p-scoring).
</Callout>

## Overview

The P2P scoring example demonstrates:

- **Scoring Profiles**: Predefined configurations for different network scenarios
- **Topic Scoring**: Advanced mesh optimization through topic-specific parameters  
- **Peer Monitoring**: Real-time tracking of peer behavior and network health
- **Configuration Management**: Flexible parameter tuning and validation

## Quick Start

```bash
# Run with default conservative scoring
cargo run --release -p example-p2p-scoring

# Try different scoring profiles
cargo run --release -p example-p2p-scoring -- --scoring-profile moderate
cargo run --release -p example-p2p-scoring -- --scoring-profile aggressive

# Enable topic scoring for enhanced performance
cargo run --release -p example-p2p-scoring -- --topic-scoring

# View detailed configuration
cargo run --release -p example-p2p-scoring -- --show-config
```

## Scoring Profiles

The example provides four predefined scoring profiles for different network scenarios:

### Off Profile
```bash
cargo run -p example-p2p-scoring -- --scoring-profile off
```

- **Use Case**: Testing, debugging, permissioned networks
- **Security**: No protection against malicious peers
- **Performance**: Higher bandwidth usage, no scoring overhead
- **Ban Threshold**: N/A (no banning)

**Suitable for:**
- Testing and development
- Permissioned networks with trusted peers
- Debugging network connectivity issues

### Conservative Profile (Default)
```bash
cargo run -p example-p2p-scoring -- --scoring-profile conservative
```

- **Use Case**: New networks, gradual scoring rollout
- **Security**: Basic protection with lenient thresholds
- **Performance**: Minimal scoring overhead
- **Ban Threshold**: -20.0 (5-minute bans)

**Suitable for:**
- New networks with limited peer diversity
- Networks prioritizing connectivity over security
- Gradual rollout of scoring mechanisms

### Moderate Profile
```bash
cargo run -p example-p2p-scoring -- --scoring-profile moderate
```

- **Use Case**: Production networks with good peer diversity
- **Security**: Balanced protection for most scenarios
- **Performance**: Optimal for general-purpose deployments
- **Ban Threshold**: -40.0 (15-minute bans)

**Suitable for:**
- Established production networks
- Networks with good peer diversity
- General-purpose deployments

### Aggressive Profile
```bash
cargo run -p example-p2p-scoring -- --scoring-profile aggressive
```

- **Use Case**: High-security or performance-critical networks
- **Security**: Strong protection against malicious behavior
- **Performance**: Higher overhead but better peer filtering
- **Ban Threshold**: -60.0 (30-minute bans)

**Suitable for:**
- High-value or security-critical networks
- Networks with many peers and good diversity
- Performance-critical applications

## Topic Scoring

Topic scoring enhances mesh performance by tracking peer behavior on specific topics:

```bash
# Enable topic scoring
cargo run -p example-p2p-scoring -- --topic-scoring
```

**Key Parameters**:
- **Time in Mesh Weight**: Rewards peers for mesh participation
- **First Message Deliveries**: Tracks unique message delivery
- **Mesh Message Deliveries**: Monitors mesh reliability
- **Invalid Message Penalties**: Punishes bad behavior

<Callout type="warning">
Topic scoring increases computational overhead but significantly improves network quality by filtering out poorly performing peers.
</Callout>

## Code Architecture

### Key Components

The example is structured around several key modules:

#### `ScoringConfig`
Central configuration management with methods to:
- Create configurations from profiles
- Generate gossipsub configurations
- Export/import configurations as JSON
- Validate parameter combinations

```rust
pub struct ScoringConfig {
    pub profile: ScoringProfile,
    pub topic_scoring: bool,
    pub ban_threshold: f64,
    pub ban_duration: Duration,
    pub mesh_config: MeshConfig,
    pub timing_config: TimingConfig,
}
```

#### `ScoringProfile`
Predefined parameter sets for different network scenarios:

```rust
pub enum ScoringProfile {
    Off,          // No peer scoring
    Conservative, // Light scoring with lenient thresholds
    Moderate,     // Balanced scoring for production
    Aggressive,   // Strict scoring for high-security
}
```

#### `NetworkMonitor`
Real-time metrics and health tracking:

```rust
impl NetworkMonitor {
    pub async fn record_block_received(&self);
    pub async fn record_peer_connected(&self, peer_id: String);
    pub async fn record_peer_banned(&self, peer_id: String, score: f64, reason: String);
    pub async fn network_health_score(&self) -> f64;
    pub async fn log_network_stats(&self);
}
```

### Integration Pattern

The example shows how to integrate scoring with kona's network components:

```rust
// Create scoring configuration
let scoring_config = ScoringConfig::new(
    ScoringProfile::Moderate,
    true, // topic_scoring
    Some(-50.0), // custom ban threshold
    Some(Duration::from_secs(1800)), // 30-minute bans
    rollup_config.block_time,
)?;

// Configure network with scoring
let network_config = NetworkConfig {
    scoring: scoring_config.peer_score_level(),
    topic_scoring: true,
    monitor_peers: scoring_config.peer_monitoring(),
    gossip_config: scoring_config.gossip_config(),
    // ... other configuration
};
```

## Security Considerations

### Profile Selection Guidelines

- **Off Profile**: Only for trusted/testing environments
- **Conservative**: Good for network establishment phase
- **Moderate**: Recommended for most production use cases
- **Aggressive**: For high-security or attack-prone networks

### Parameter Tuning Best Practices

- **Lower ban thresholds** increase security but may cause isolation
- **Shorter ban durations** allow faster peer recovery
- **Topic scoring** improves quality but increases computational overhead
- **Monitor ban rates** to avoid excessive peer exclusion

<Callout type="warning">
Overly aggressive scoring parameters can lead to network partitioning. Always monitor peer connectivity and ban rates when tuning parameters.
</Callout>

## Performance Tuning

### Mesh Size Configuration

Different profiles use different mesh parameters optimized for their use cases:

```rust
// Conservative: smaller mesh, lower overhead
MeshConfig {
    mesh_n: 8,
    mesh_n_low: 6,
    mesh_n_high: 12,
    gossip_lazy: 6,
    flood_publish: false,
}

// Aggressive: larger mesh, better redundancy  
MeshConfig {
    mesh_n: 12,
    mesh_n_low: 10,
    mesh_n_high: 18,
    gossip_lazy: 10,
    flood_publish: false,
}
```

### Timing Parameters

Standard timing configuration works for most networks:

```rust
TimingConfig {
    heartbeat_interval: Duration::from_millis(500),
    fanout_ttl: Duration::from_secs(60),
    history_length: 12,
    history_gossip: 3,
}
```

## Command Line Reference

```
Options:
  -c, --l2-chain-id <L2_CHAIN_ID>
          The L2 chain ID to use [default: 10]

  -g, --gossip-port <GOSSIP_PORT>
          Port to listen for gossip on [default: 9099]

  -d, --disc-port <DISC_PORT>
          Port to listen for discovery on [default: 9098]

  -i, --discovery-interval <DISCOVERY_INTERVAL>
          Interval to send discovery packets in seconds [default: 5]

  -s, --scoring-profile <SCORING_PROFILE>
          Scoring profile: off, conservative, moderate, aggressive [default: conservative]

      --topic-scoring
          Enable topic scoring for enhanced mesh performance

      --ban-threshold <BAN_THRESHOLD>
          Threshold below which peers are banned (negative values)

      --ban-duration <BAN_DURATION>
          Duration for which banned peers remain banned

      --monitoring
          Enable network monitoring and metrics collection [default: true]

      --show-config
          Print detailed scoring configuration and exit

  -v, --verbosity <VERBOSITY>
          Verbosity level [default: 0]
```

## Troubleshooting

### Common Issues

#### No Peers Connecting
```bash
# Try disabling scoring for debugging
cargo run -p example-p2p-scoring -- --scoring-profile off --verbosity 2

# Check firewall and network connectivity
# Verify bootstrap nodes are reachable
```

#### Excessive Peer Bans
```bash
# Use more lenient profile
cargo run -p example-p2p-scoring -- --scoring-profile conservative

# Or increase ban threshold (less negative)
cargo run -p example-p2p-scoring -- --ban-threshold -10.0

# Disable topic scoring temporarily
cargo run -p example-p2p-scoring -- --scoring-profile moderate
```

#### Poor Network Performance
```bash
# Enable topic scoring for better mesh
cargo run -p example-p2p-scoring -- --topic-scoring

# Use more aggressive profile for better filtering
cargo run -p example-p2p-scoring -- --scoring-profile aggressive

# Monitor for network partitioning
cargo run -p example-p2p-scoring -- --verbosity 2
```
